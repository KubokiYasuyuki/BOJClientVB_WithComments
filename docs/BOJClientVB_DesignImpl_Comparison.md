## 4. `ReadRegistryValues`（レジストリ値読み込み）

### 4.1 判定
🔴 **乖離（要対応）**

### 4.2 対応箇所
- 設計：ID-05 §2.2.1「ホスト情報（ホスト名）と**ポート番号**を読み」「タイムアウト値はレジストリで設定可能」
- 実装：<ref_snippet file="/home/ubuntu/work/sample/BOJClientVB/BOJClientVBCLS.vb" lines="112-136" />

### 4.3 差分
| 項目 | 設計書 | 実装（値名） | 実装（型） |
|---|---|---|---|
| 1番目ホスト名 | 「ホスト情報（ホスト名）」（値名未定義） | `Host01` | REG_SZ |
| 2番目ホスト名 | 同上 | `Host02` | REG_SZ |
| ポート情報 | 「**ポート番号**」 | `ServiceName`（文字列） | REG_SZ |
| タイムアウト | 「タイムアウト値」（単位未定義） | `SocketTimeOut`（秒） | REG_DWORD |

### 4.4 主要な乖離
1. **ポート情報の表現方法が異なる**
   - 設計は「ポート番号」（数値を直接レジストリに書く前提）
   - 実装は「サービス名」（services ファイルからポート番号を引く）
   - 結果：`ServiceName` 値が services ファイルに登録されていない場合、コード側は 50000 にフォールバックするが、**設計書の文面では 50000 フォールバックは想定されていない**
2. **値名・型・単位が設計書に一切記載されていない**
   - 第三者が設計書のみで実装を再現できない
3. **`SocketTimeOut` の適用範囲が未記載**（接続タイムアウトではなく受信/送信タイムアウトのみに効く）

### 4.5 VC6 との関係
- VC6 版も同じ値名（`Host01`/`Host02`/`ServiceName`/`SocketTimeOut`）を使用
- つまり実装は VC6 から一貫しているが、**設計書が VC6 時代から値名を明記してこなかった** ことに起因する積年の乖離

### 4.6 改善提案
**案A（推奨：設計書更新）**
- 設計書 §2.2.1 または §3.1 に以下を明記
  - レジストリパス（`HKLM\SOFTWARE\sys148\Profiles`）
  - 値名・型・用途一覧（Host01 / Host02 / ServiceName / SocketTimeOut）
  - 50000 フォールバック仕様
- 具体文案は前回メッセージ「3.1.1 レジストリ設定」案を参照
- **現ユーザ方針では追加しない選択となったため、本論点は「認識共有のみ」で保留**

**案B（コード変更）**
- 設計書の「ポート番号」に忠実に合わせ、レジストリから直接ポート番号を読む仕様に変更（`ServiceName` → `PortNumber`）
- VC6 互換性を失うため非推奨

**本レポートの推奨判断**：案A（ただしお客様意向で現状維持）

- 実装：<ref_snippet file="/home/ubuntu/work/sample/BOJClientVB/BOJClientVBCLS.vb" lines="365-367" />

### 5.3 確認結果
| 観点 | 設計書 | 実装 |
|---|---|---|
| ヘッダー | `VIDR` | `VIDR` |
| 区切り | 空白1文字以上 | 空白1文字 |
| 引数 | `DUMMY, DUMMY` | `{user}, {pass}`（実呼出は "DUMMY", "DUMMY"） |
| 改行 | 0x0A | `vbLf` (= Chr(10) = 0x0A) |

### 5.4 補足・軽微な論点
- 設計書の例示行には `VIDR DUMMY, DUMMY` の形式でカンマの後に空白が1つあるが、`[空白文字]+` 規定はヘッダーと引数の間のみを規定しており、引数内のカンマ前後空白は未規定。実装は `", "`（カンマ＋空白）で送信しているので、サーバ側がこれを受容することを前提とする
- サーバ側パーサがカンマのみ／カンマ+空白の両方を受容する実装になっているかは別途確認推奨（動作しているなら問題なし）

### 5.5 改善提案
- コード変更：不要
- 設計書更新：任意。「引数のカンマ区切りは `, `（カンマ+空白1文字）とする」等を §5.2.1 に補足すると精度が上がる

## 6. `AnalyseResVIDR`（VIDR 応答解析）

### 6.1 判定
🟡 **乖離（軽微）**

### 6.2 対応箇所
- 設計：ID-05 §5.2.1「応答電文: ACK DBアクセスID,DBMSパスワード,オンライン開局状態」「NG」「オンライン開局は文字の1または0で返され、1が開状態」
- 実装：<ref_snippet file="/home/ubuntu/work/sample/BOJClientVB/BOJClientVBCLS.vb" lines="384-417" />

### 6.3 確認結果
| 観点 | 設計書 | 実装 | 判定 |
|---|---|---|---|
| 正常応答ヘッダー | `ACK` | `ACK` で始まる場合のみ解析続行 | 🟢 |
| 異常応答 | `NG` を別途想定 | `NG` も「ACK でない＝解析失敗(False)」扱い | 🟡 |
| 引数区切り | カンマ | カンマ | 🟢 |
| カンマ前後の空白 | 応答形式定義にはなし、例示には空白あり | `Split(",") + Trim()` で両方受容 | 🟢 |
| オンラインフラグ | 1 または 0 | `Integer.TryParse`、失敗時は 0 | 🟢（異常値は 0 に倒れる） |
| 改行処理 | LF 終端 | `IndexOf(vbLf)` で除去 | 🟢 |

### 6.4 乖離点の詳細
- **"NG" 応答の扱い**：設計書は「NG」を正式応答として定義しているが、実装では「ACK でない」全般と同一扱いで `False` を返し、呼び出し元で `szRET="3"`（その他のエラー）＋ `szERR="NG"` として処理される
- 厳密には「NG」は **サーバ側が明示的に拒否した** 応答であり、ネットワーク障害や応答形式破綻とは区別すべき
- しかし設計書上 "NG" を受け取った時の szRet 値は定義されていない（`szRet` の値定義に「認証拒否」は無く「1〜4」のみ）
- ⇒ 実装側を変えるよりも、設計書で `"NG"` 応答時の szRet 仕様を明文化する余地あり（例：`"3"` として扱う、の形で後付け明記）

### 6.5 改善提案
- コード変更：不要
- 設計書更新（任意）：§3.2.1 szRet 説明に「サーバから `NG` 応答を受けた場合は `"3"` を返し、`szError` に `"NG"` を格納する」等を追記
- お客様方針（細部不関知）ならば現状維持で支障なし

## 7. `GetServicePort`（サービス名→ポート番号変換）

### 7.1 判定
⚪ **設計外（設計書に言及なし）**

### 7.2 対応箇所
- 設計：言及なし
- 実装：<ref_snippet file="/home/ubuntu/work/sample/BOJClientVB/BOJClientVBCLS.vb" lines="434-472" />
- VC6 原典：`CONNECTSOCKET.CPP:27` `getservbyname(lpszService, "tcp")`（Winsock API 経由）

### 7.3 VC6 との挙動差
| 観点 | VC6 | VB.NET |
|---|---|---|
| services ファイル参照 | `getservbyname`（Winsock API） | `%SystemRoot%\System32\drivers\etc\services` を自前でパース |
| 解決失敗時の挙動 | `Create()` が FALSE を返し、コネクション失敗扱い（szRet="2"）| `-1` を返し、呼び出し元で 50000 にフォールバック |
| tcp/udp 区別 | `"tcp"` 指定で tcp のみ | プロトコル区別なし（"/tcp" / "/udp" 等の後ろ部分は無視） |

### 7.4 差分
- **50000 フォールバック** は VB.NET で独自追加された挙動（VC6 には無い）
- 設計書 §5.1 に「サービスポートは50000とする」との明記があるため、結果的に 50000 固定で稼働する方向であれば **設計書と整合的**
- 逆に VC6 では services ファイルに `aauth 50000/tcp` 登録が前提であり、登録漏れ環境では接続不能
- したがって VB.NET 側の方が設計 §5.1 に忠実。ただし `services` ファイルを参照する手順自体は設計書に書かれていない

### 7.5 改善提案
- **案A（推奨）**：設計書 §3.1 または §5.1 に「サービス名 `aauth` から services ファイルでポート番号を解決する。解決できない場合は設計規定の 50000 を使用する」旨を一文追記
- **案B**：services 参照ロジック自体を削除し、常に 50000 固定とする（設計 §5.1 にもっと忠実）
  - ただし運用面で柔軟性を失うため非推奨
- **案C（現状維持）**：お客様方針どおり何もしない。50000 で稼働すると確認済みであれば実害なし

## 11. 横断的な所見

### 11.1 設計書側で未記載となっている主要項目
以下は実装にあるが設計書に記述がない項目。重要度順：

1. **(高)** レジストリ値の詳細仕様（値名・型・単位） ← §4 の論点
2. **(中)** COM GUID 互換性維持（VC6 OCX との同一 GUID 再利用） ← §8 の論点
3. **(中)** `services` ファイル参照と 50000 フォールバック挙動 ← §7 の論点
4. **(中)** x64 固定の明記 ← §9 の論点
5. **(低)** `Profiles` サブキー優先の目的 ← §3 の論点
6. **(低)** COM インスタンス化時のレジストリ自動読込 ← §2 の論点
7. **(低)** `SocketTimeOut` の適用範囲（受信のみ / 接続には適用されない） ← §4 の論点
8. **(低)** `NG` 応答時の szRet マッピング ← §6 の論点

### 11.2 実装側で修正済み／修正不要と判断した項目
- `AAVerifyID` の szRet 値（PR #1 で設計準拠）
- 未使用 DES/Base64 領域（PR #1 で削除）
- 戻り値「常に 0」仕様（設計書側で明文化、VC6 踏襲）

### 11.3 現実装の強み
- VC6 OCX との **GUID 互換性** が維持されており、呼び出し側 VBA のコード改変を最小化
- `Profiles` キー優先機構により複数環境切替に備えた構造（設計未記載だが柔軟性あり）
- 例外時には必ず `szRet` にステータスを設定して戻すため、**呼び出し側は戻り値ではなく szRet で判定する** という設計原則が一貫している

### 11.4 現実装のリスク要因
- 設計書に値名・型・フォールバック挙動が明記されていないため、第三者が設計書だけで実装・検証できない
- `OptionStrict Off` のまま運用されており、レジストリ値の型不正で例外が発生する経路がある
  - 例：`SocketTimeOut` が REG_SZ で書かれていた場合、`Convert.ToInt32(Object)` は `String→Integer` 変換に成功することもあり、想定外の挙動になり得る
